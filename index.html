0A01 — Home Lab 01

Ambiente aziendale simulato · Debian 13 Trixie · Windows Server DC · Docker · WireGuard

Premessa

Sono Alessandro Muzi, ho recentemente completato un corso di 300 ore presso il CEFI, focalizzato sulla gestione dei sistemi e sulla sicurezza informatica.

Questo repository nasce con l'obiettivo di documentare la creazione di un ambiente aziendale simulato in cui il Domain Controller Windows gestisce Active Directory e DNS, mentre i sistemi Debian ospitano tutti i servizi infrastrutturali complementari.

La struttura è la seguente: il server Debian 13 "Trixie" (Oracle VM) funge da core dell'infrastruttura, configurato con funzioni di routing e NAT tramite iptables e IP forwarding, oltre alla gestione del servizio DHCP Kea. Il server ospita 3 interfacce di rete: una in modalità bridge (enp0s3) e due LAN dedicate (LAN1=enp0s8, LAN2=enp0s9).

La prima LAN (10.0.10.1) è destinata a due client Linux (Fedora 10.0.10.101 e Ubuntu 10.0.10.102), mentre la seconda (10.0.20.1) collega il server Debian alla macchina Windows Server (Controller di Dominio), che gestisce Active Directory e il servizio DNS per l'intera struttura. Tutto il traffico della rete privata transita crittografato all'interno del tunnel WireGuard (VPN).

Samba, installato su Debian, si interfaccia con AD tramite winbind e gestisce un Print Server condiviso via CUPS. Lo stack di monitoring composto da Grafana Alloy, Prometheus, Loki e cAdvisor, containerizzato in Docker, si occupa del monitoraggio e del logging dello stato delle macchine e della rete. Infine, Rsync e Cron gestiscono in modo incrementale il backup automatizzato dell'intero ambiente tramite script dedicati.

Interfaces & Name Resolution

Il gateway Debian espone tre interfacce di rete: una interfaccia WAN configurata in modalità bridge per l'accesso esterno e due interfacce LAN dedicate ai segmenti di rete interni.

/etc/network/interfaces

# Include eventuali file di configurazione aggiuntivi dalla directory interfaces.d/
source /etc/network/interfaces.d/*
# Interfaccia di loopback (localhost)
auto lo
iface lo inet loopback
# --- WAN (Bridge VirtualBox) ---
# Configurazione statica per evitare interferenze di systemd-networkd
auto enp0s3
iface enp0s3 inet static
  address         192.168.1.4/24
  gateway         192.168.1.1
  # DNS di fallback pubblici (il DNS primario 10.0.20.100 verrà configurato in resolv.conf)
  dns-nameservers 1.1.1.1 8.8.8.8
# --- LAN1 (Client Linux: Fedora 10.0.10.101, Ubuntu 10.0.10.102) ---
auto enp0s8
iface enp0s8 inet static
  address         10.0.10.1/24
# --- LAN2 (Windows Server / Domain Controller / DNS 10.0.20.100) ---
auto enp0s9
iface enp0s9 inet static
  address         10.0.20.1/24

/etc/resolv.conf

Nota di Protezione: Il file è stato reso esplicitamente immutabile tramite il comando chattr +i /etc/resolv.conf. Questo accorgimento impedisce a demoni di sistema come systemd-resolved o client DHCP esterni di sovrascrivere o alterare i puntamenti DNS al riavvio del server.

search lab.local
# DNS primario: Domain Controller Windows (autoritativo per lab.local e Active Directory)
nameserver 10.0.20.100
# DNS di fallback pubblici: usati se il DC non è raggiungibile

nameserver 1.1.1.1

DHCP — Kea Server

Il server DHCP Kea gestisce in modo centralizzato il rilascio degli indirizzi IP per entrambe le reti interne (LAN1 e LAN2), occupandosi di distribuire i corretti parametri di rete, i gateway e i puntamenti DNS.

Iniezione DNS & Integrazione AD: A tutti i client di rete viene iniettato automaticamente l'IP 10.0.20.100 come server DNS primario. Questo garantisce che qualsiasi macchina agganciata alle LAN sia in grado di risolvere nativamente il dominio Active Directory lab.local.
Routing Avanzato (Classless Static Routes): È stata implementata l'opzione DHCP classless-static-route codificata in esadecimale raw. Questa istruisce direttamente la tabella di routing dei client, forzando il passaggio del traffico destinato alla sottorete VPN WireGuard attraverso gli IP interni del gateway Debian.

Firewall — iptables legacy & nftables

La sicurezza perimetrale e il routing interno sono gestiti tramite uno script iptables nativo. Il firewall implementa una politica di tipo Whitelisting (DROP di default), isola e unisce selettivamente i segmenti di rete e attiva il NAT (Masquerading) per l'uscita su Internet.

Script di Configurazione (/etc/systemd/system/nat.sh)

#!/bin/bash

# Svuota le regole preesistenti e azzera le tabelle di NAT
iptables -F
iptables -t nat -F

# Consente tutto il traffico interno sull'interfaccia di loopback
iptables -A INPUT -i lo -j ACCEPT

# Mantiene attive le connessioni in entrata già stabilite o correlate (Stateful Inspection)
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Servizi esposti sul Gateway: SSH (22), WireGuard VPN (51820) e protocollo diagnostico ICMP
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -p udp --dport 51820 -j ACCEPT
iptables -A INPUT -p icmp -j ACCEPT

# Configura il NAT (Masquerading) sull'interfaccia WAN per consentire l'uscita a Internet
iptables -t nat -A POSTROUTING -o enp0s3 -j MASQUERADE

# Consente il transito dei pacchetti per tutte le connessioni già stabilite
iptables -A FORWARD -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Permette l'inoltro del traffico dalla VPN verso le reti interne LAN1, LAN2 e la WAN
iptables -A FORWARD -i wg0 -o enp0s9 -j ACCEPT
iptables -A FORWARD -i wg0 -o enp0s8 -j ACCEPT
iptables -A FORWARD -i wg0 -o enp0s3 -j ACCEPT

# Gestisce lo stato delle connessioni di ritorno destinate alla VPN
iptables -A FORWARD -o wg0 -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Instrada il traffico in uscita verso Internet per i client di LAN1 e LAN2
iptables -A FORWARD -i enp0s8 -o enp0s3 -j ACCEPT
iptables -A FORWARD -i enp0s9 -o enp0s3 -j ACCEPT

# Abilita la comunicazione bidirezionale diretta tra i segmenti di rete LAN1 e LAN2
iptables -A FORWARD -i enp0s9 -o enp0s8 -j ACCEPT
iptables -A FORWARD -i enp0s8 -o enp0s9 -j ACCEPT

# Imposta le policy di Whitelisting: blocca tutto il traffico non esplicitamente autorizzato
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Abilita il Packet Forwarding a livello di kernel Linux
echo 1 > /proc/sys/net/ipv4/ip_forward

Esecuzione automatica dello script (/etc/systemd/system/firewall.service)

[Unit]
# Metadati del servizio e dipendenze di avvio
Description = Firewall iptables - NAT e regole di rete
After       = network.target

[Service]
ExecStart   = /etc/systemd/system/nat.sh

[Install]
# Assicura che il firewall parta automaticamente in modalità multi-utente
WantedBy    = multi-user.target

VPN — WireGuard

WireGuard è installato su tutti i nodi dell'infrastruttura. L'interfaccia virtuale wg0 ascolta sulla porta UDP 51820.

Le chiavi private e pubbliche reali sono state oscurate (*) per motivi di sicurezza.

wg0.conf — Configurazione del Server (Gateway Debian)

[Interface]
# Indirizzo IP e subnet dedicati alla rete privata virtuale (VPN)
Address    = 10.8.0.1/24
# Disabilita il salvataggio automatico delle modifiche apportate a runtime via CLI
SaveConfig = False
# Regole iptables applicate all'avvio: abilitano l'inoltro e il NAT
//PostUp     = iptables -A FORWARD -i wg0 -j ACCEPT;
#                    iptables -A FORWARD -o wg0 -j ACCEPT;
#                    iptables -t nat -A POSTROUTING -o enp0s3 -j MASQUERADE
// Rimuove le regole iptables dedicate alla VPN al momento dello spegnimento
//PostDown   = iptables -D FORWARD -i wg0 -j ACCEPT;
#                    iptables -D FORWARD -o wg0 -j ACCEPT;
#                    iptables -t nat -D POSTROUTING -o enp0s3 -j MASQUERADE
# Porta UDP di ascolto principale su cui il server attende le connessioni in ingresso
ListenPort = 51820
# Chiave privata crittografica del server gateway
PrivateKey = *

[Peer]
# Chiave pubblica identificativa del primo client (es. Domain Controller Windows)
PublicKey  = *
# Indirizzo IP statico univoco assegnato al primo peer all'interno del tunnel
AllowedIPs = 10.8.0.2/32

[Peer]
# Chiave pubblica identificativa del secondo client (es. Client Linux Fedora)
PublicKey  = *
# Indirizzo IP statico univoco assegnato al secondo peer all'interno del tunnel
AllowedIPs = 10.8.0.3/32

[Peer]
# Chiave pubblica identificativa del terzo client (es. Client Linux Ubuntu)
PublicKey  = *
# Indirizzo IP statico univoco assegnato al terzo peer all'interno del tunnel
AllowedIPs = 10.8.0.4/32

Servizi di Directory e Risoluzione dei Nomi (Active Directory & DNS)

Questa sezione documenta l'infrastruttura di identità e risoluzione dei nomi del laboratorio, centralizzata su un Domain Controller Windows Server (IP statico: 10.0.20.100). L'estrazione dei dati e la manutenzione ordinaria del server sono state eseguite sfruttando la catena di accesso remota e i tunnel VPN/SSH dal Gateway Debian, una configurazione ideale per testare l'interoperabilità e la robustezza della rete. Il flusso si sviluppa a partire dalla Macchina fisica:

Macchina fisica (CMD) ➔ SSH su Fedora/Debian ➔ Cockpit (Browser) ➔ SSH verso il DC Windows ➔ PowerShell

Architettura del Dominio e Unità Organizzative (AD DS)

Il cuore della gestione delle identità è un dominio Active Directory denominato lab.local. La verifica dei parametri logici della foresta e la mappatura strutturale delle risorse (Utenti, Gruppi e Computer) mostrano un'alberatura logica pulita e l'adozione di livelli funzionali moderni.

Il livello di funzionalità del dominio e della foresta è impostato su Windows Server 2016: questo rappresenta il livello logico standard e ottimale introdotto da Microsoft, garantendo la totale compatibilità con le funzionalità moderne di Active Directory anche nel caso di aggiornamenti o edizioni successive di Windows Server.

Il Domain Controller agisce come server DNS primario e autoritativo per l'intera infrastruttura. Come documentato nella sezione precedente, questo specifico puntamento viene iniettato automaticamente a tutti i client di rete tramite il server Kea DHCP. Analisi Tecnica del Flusso DNS:
  • Integrazione nativa in AD (IsDsIntegrated = True): Le zone vitali per l'infrastruttura, come lab.local e la zona speciale _msdcs.lab.local (fondamentale per i record SRV di localizzazione dei servizi di dominio), sono integrate nel database di Active Directory, garantendo aggiornamenti dinamici protetti e massima sicurezza.
  • Risoluzione Inversa Attiva: È stata configurata la zona di reverse lookup 10.0.10.in-addr.arpa per la subnet dei client Linux. Si tratta di una best practice indispensabile per permettere ai servizi Linux e agli strumenti di monitoring di risolvere gli indirizzi IP in nomi host reali.
  • Interoperabilità del Gateway: La presenza di una zona primaria dedicata esplicitamente a debian dimostra la perfetta federazione, la raggiungibilità reciproca e il corretto scambio di record tra il server Linux e l'ambiente Microsoft Windows.

Configurazione Samba (/etc/samba/smb.conf)

[Global]
## Forza la shell predefinita per gli utenti di dominio che effettuano l'accesso
template shell = /bin/bash

## Nome NetBIOS del dominio Active Directory
workgroup = LAB

## Imposta Samba in modalità membro nativo di un dominio Active Directory (Active Directory Services)
security = ADS

## Specifica il Realm Kerberos in lettere MAIUSCOLE, indispensabile per il funzionamento del trust ticket
realm = LAB.LOCAL

## Integrazione del sottosistema di stampa (CUPS) per la condivisione di stampanti in rete
printing = cups
printcap name = cups
load printers = yes

## Permette di omettere il prefisso del dominio (es. 'utente' invece di 'LAB\utente') nei comandi Linux
winbind use default domain = yes

## Abilita l'enumerazione completa degli utenti e gruppi AD tramite winbind.
## Necessario per la risoluzione locale con getent, ma sconsigliato in domini con molti oggetti
winbind enum users = yes
winbind enum groups = yes
winbind expand groups = 4

## Configurazione del backend IDMAP per la mappatura dei SID di Windows in UID/GID di Linux
## Definisce l'intervallo di allocazione degli ID locali (da 3000 a 79999) tramite database TDB locale
idmap config * : backend = tdb
idmap config * : range = 3000-79999

[Printers]
## Configurazione di default per l'esposizione automatica delle stampanti di sistema
comment = All Printers
path = /var/spool/samba
browseable = yes
printable = yes
guest ok = yes
read only = yes

[Print$]
## Area di storage per i driver delle stampanti condivise Windows
path = /var/lib/samba/printers
browseable = yes
read only = yes

[ProgettiDev]
## --- CONDIVISIONE IBRIDA APPLICATIVA ---
## Percorso fisico sul file system locale di Debian
path = /srv/samba/progetti_dev
## SBARRAMENTO DI SICUREZZA: Solo gli utenti appartenenti al gruppo globale "G_Sviluppatori"
## censito dentro Windows Active Directory possono accedere a questa cartella.
valid users = @"LAB\G_Sviluppatori"
read only = no
guest ok = no

[dev_share]
## --- Area di storage condivisa per l'ambiente di sviluppo e testing interno ---
path = /srv/samba/dev
read only = no
browseable = yes
valid users = @smb_dev_share
## Forza l'appartenenza dei file creati al gruppo di sviluppo locale
force groups = smb_dev_share
## Maschere restrittive sui permessi (POSIX): File in 660, directory con Sticky Bit e SGID in 2770
create mask = 0660
directory mask = 2770

Docker & Monitoring

Stack di monitoraggio containerizzato via Docker Compose. Prometheus e cAdvisor raccolgono metriche, Loki e Grafana Alloy gestiscono i log, Grafana li visualizza sulla porta 3000.

GRAFANA ALLOY (config.alloy)

// Scopre automaticamente tutti i container Docker attivi tramite il socket
discovery.docker "docker_containers" {
  host = "unix:///var/run/docker.sock"
}

// Pulisce i metadati Docker e assegna la label service_name
discovery.relabel "docker_targets" {
  targets = discovery.docker.docker_containers.targets
  rule {
    source_labels = ["__meta_docker_container_name"]
    regex          = "/(.+)"
    target_label   = "service_name"
  }
}

// Raccoglie i log dallo stdout/stderr di tutti i container scoperti
loki.source.docker "docker_logs" {
  host       = "unix:///var/run/docker.sock"
  targets    = discovery.docker.docker_containers.targets
  forward_to = [loki.write.local_loki.receiver]
}

// Parsing automatico dei log in formato Docker JSON:
// stage.docker estrae timestamp, stream (stdout/stderr) e messaggio
// e corregge il fuso orario per la visualizzazione in Grafana
loki.process "parse_docker_timestamp" {
  stage.docker {}
  forward_to = [loki.write.local_loki.receiver]
}

  // Spinge i log raccolti verso Loki tramite la rete interna Docker
loki.write "local_loki" {
  endpoint {
    url = "http://loki:3100/loki/api/v1/push"
  }
}

Backup & Cron

Gestione automatizzata dei backup per le share Samba tramite rsync e pianificazione periodica via cron. Il sistema implementa una politica di rotazione dei dati per mantenere uno storico di 7 giorni.

Script di Backup (/usr/local/bin/backup-samba.sh)

#!/bin/bash
set -euo pipefail

# Sorgente
SOURCE="/srv/samba/"
# Destinazione locale
BACKUP_ROOT="/backup/samba"
# Numero backup da mantenere
RETENTION=7
# Data backup
DATE=$(date +"%Y-%m-%d")
# Cartella backup corrente
DEST="${BACKUP_ROOT}/backup_${DATE}"
LOGFILE="/var/log/backup-samba.log"

# Crea directory root se non esiste
mkdir -p "${BACKUP_ROOT}"

log () {
    echo "$(date '+%Y-%m-%d %H:%M:%S') - $1" >> "${LOGFILE}"
}

log "___Avvio Backup__"

# Backup rsync
#-aAXv:      -a: modalità archivio, preserva permessi, ownership, timestamp, symlink, dir ricorsive
#            -A: preserva ACL
#            -X: preserva extended attributes
#            -v: verbose
## --delete rimuove dalla destinazione i file che non esistono più nella sorgente.
if rsync -aAXv --delete "${SOURCE}" "${DEST}" >> "${LOGFILE}" 2>&1; then
     log "SUCCESSO: Backup completato in ${DEST}"
     else
     log "ERRORE: Backup fallito"
     exit 1
fi

# Rotazione: elimina backup più vecchi
cd "${BACKUP_ROOT}" || exit 1

# 1. ls -1dt */  -> Elenca le cartelle (-d) su una singola colonna (-1) ordinate dalla più recente alla più vecchia (-t)
# 2. tail -n +8 -> Salta le prime 7 cartelle (i backup da tenere) e prende solo quelle dall'ottava in poi
OLD_BACKUPS=$(ls -1dt backup_* 2>/dev/null | tail -n +$((RETENTION + 1)))

for OLD in ${OLD_BACKUPS}; do
    log "Elimino vecchio backup: ${OLD}"
    rm -rf "${OLD}"
done

log "__Fine backup__"

Pianificazione automatica (crontab)

# crontab -e
0 2 * * * /usr/local/bin/backup-samba.sh >> /var/log/backup-samba.log 2>&1
